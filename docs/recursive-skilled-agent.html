RecursiveSkilledAgent

The primary entry point for running structured, skill-bound work without drifting into open-ended guessing.

Vision

RecursiveSkilledAgent is a task-focused agent that exists to execute repeatable work through explicit skills, not to behave like a universal assistant. It treats skill definitions as the source of truth and keeps the model inside those boundaries, which reduces hallucination risk and keeps outputs predictable. Instead of opportunistically deciding when to use external help, it assumes that work is already captured as skills and coordinates those skills directly.

This framing matters for production systems. When a task is repeated, it should be captured as a skill and executed in a controlled loop, not re-invented on every prompt. The agent therefore optimizes for consistency and auditability rather than broad, speculative knowledge.

What It Does

At runtime the agent discovers skills on disk, registers them, and routes a user request to the most appropriate skill or orchestration flow. When a request requires multiple steps, it delegates to orchestration skills that run an agentic loop (ReAct-style or Soplang-based planning) and decide which skill to execute next until the user goal is satisfied. You can learn how those sessions work in the Agentic Sessions page.

This means the agent remains lightweight and deterministic in the core, while complex, multi-step reasoning lives inside purpose-built skills and session types. You keep the flexibility of an LLM with the guardrails of an explicit workflow.

Discovery Model

Skill discovery is filesystem-driven. The agent scans for skills/ directories, parses skill descriptors, and registers them with a canonical name. It can search upwards from a starting directory (default) or search downwards by locating a repos root when configured to scan in that direction. 

workspace/
├── skills/
│   ├── reporting/
│   │   └── daily-summary/
│   │       └── oskill.md
│   ├── inventory/
│   │   └── equipment/
│   │       └── tskill.md
│   └── tools/
│       └── data-normalizer/
│           └── cskill.md
└── app/
    └── src/

The agent also includes built-in skills shipped with the library and can register additional skill roots you provide. Internal skills are marked as built-in and can be hidden from LLM selection while remaining callable by name. The organization and taxonomy of skills is explained in Subsystems and Skills.

Execution Philosophy

The agent does not attempt to answer arbitrary questions. It is intentionally constrained to what your skills can do. If a request is ambiguous or outside the skill boundary, the system expects either a specialized skill or an explicit escalation path, not a speculative free-form answer.

This is also why it supports recursive execution: a skill can call other skills, which can then call further skills. The recursion is governed by the agent’s registry and by allowlists in orchestration or MCP skills, so it remains bounded.

Key Responsibilities

RecursiveSkilledAgent coordinates a small set of specialized services rather than doing everything itself. Skill discovery, selection, and execution are delegated to dedicated modules so the agent can remain a stable entry point while the system evolves.

  • Discovers skills and registers canonical names, aliases, and metadata.
  • Routes requests to subsystems based on skill type.
  • Injects session memory and I/O adapters into execution context.
  • Manages lifecycle callbacks and cancellation signals.

Session Memory and Safety

The agent owns session memory because state is essential for iterative workflows. It maintains a default session for CLI usage and supports multiple session IDs for webchat or multi-tenant contexts. Session memory is injected automatically into calls unless you override it explicitly, which keeps skill implementations clean and deterministic.

Session storage is guarded by configurable TTL and LRU eviction. A background cleanup timer can evict idle sessions, and the agent exposes helpers for manual cleanup and graceful shutdown.

Built-In Skill Handling

The agent always registers built-in skills from the package’s internal skills/ directory. It also ensures that mirror-code-generator is registered first, because it is used to generate executable code for code skills. When a cskill is discovered, the agent schedules a generation pass through mirror-code-generator automatically. The details of built-in skills are documented in Built In Skills.

Configuration Surface

RecursiveSkilledAgent can be configured to fit strict production environments or exploratory development. It accepts an existing LLMAgent or creates one for you, allows you to control search boundaries, and wires in I/O and reporting adapters for interactive runtimes. 

const agent = new RecursiveSkilledAgent({
    startDir: process.cwd(),
    searchUpwards: true,
    additionalSkillRoots: ['./team-skills'],
    exposeInternalSkills: false,
    tierConfig: { plan: 'plan', execution: 'fast', code: 'code' },
    sessionConfig: { maxSessions: 1000, sessionTTL: 7_200_000, cleanupInterval: 300_000 },
});

The tierConfig is intentionally explicit. Planning and execution are treated as distinct tiers, and the agent can override skill-level tiers for internal orchestration, which is essential when you want fast decision-making but deeper execution for specific skill families.

Canonical Execution Flow

user prompt
   ↓
RecursiveSkilledAgent
   ├─ discover & register skills
   ├─ select orchestrator or direct skill
   ├─ inject session memory & IO
   └─ delegate to subsystem
          ↓
     skill execution / agentic session
          ↓
       final response

The central promise is that every decision and action routes through a named skill. The agent’s job is not to be omniscient, but to be reliable and explainable.
