LLMAgent

The foundational interface for all LLM communication and agentic session orchestration.

Vision

LLMAgent is the single, consistent gateway to a language model. It exists so every skill, subsystem, and agent uses the same execution surface, which keeps model behavior uniform across the library. Instead of allowing each skill to implement its own LLM calls, LLMAgent provides a shared interface for prompts, structured responses, and session-based planning.

This design keeps LLM access explicit and measurable. It becomes clear when, why, and how a model is invoked, and that clarity is essential when you want predictable behavior rather than ad‑hoc experimentation.

What It Does

LLMAgent exposes a small set of core methods for direct prompt execution and for starting agentic sessions. It can return plain text, structured JSON, or code payloads, and it normalizes these response types so skills do not re-implement parsing logic. The same instance is reused by the RecursiveSkilledAgent and by every skill, ensuring uniform model behavior across the entire system.

Direct Prompt Interface

The direct prompt API is intentionally minimal. It accepts a prompt string and options for tier selection, response shape, and execution context. This keeps the interface raw and composable, which is important for skills that need precise control over how the model is used. 

const result = await llmAgent.executePrompt('Summarize the report', {
    tier: 'fast',
    responseShape: 'text',
    context: { intent: 'summarize-report' },
});

When the response shape is JSON or code, LLMAgent validates and normalizes the output so the caller can rely on a consistent structure. This is a key reason the library recommends that all LLM interactions go through this class.

Core APIs and How They Differ

LLMAgent exposes a small number of methods so every skill interacts with the model the same way. The differences are intentional and map to distinct interaction patterns rather than to different model providers.

  • executePrompt(promptText, options) is the primary, structured entry point. It supports response shaping (text, JSON, code) and is the default for skill execution.
  • complete(options) is a lower-level completion helper used internally or for advanced flows. It provides fewer guarantees about response shape.
  • startLoopAgentSession(tools, initialPrompt, options) starts a ReAct-style loop session for iterative multi-step reasoning.
  • startSOPLangAgentSession(skillsDescription, initialPrompt, options) starts a Soplang session that generates and executes a deterministic plan.
  • detectIntents(skillsDescription, userPrompt, options) performs intent classification against a provided skill list.
  • cancel() stops in-flight requests and clears the current processing callbacks.
  • getCallLog() returns per-call telemetry such as tier, model, and latency.

In practice, executePrompt is the method most skills use. The session APIs are reserved for orchestration skills or for custom flows where planning and execution need to be separated.

Agentic Sessions

LLMAgent is also responsible for creating and running agentic sessions. A Loop session is used for ReAct-style iteration, while a Soplang session produces and executes a deterministic plan. These session types are described in depth on the Agentic Sessions page and the Soplang Plan page. 

const loopSession = await llmAgent.startLoopAgentSession(skills, 'Start the workflow');

const sopSession = await llmAgent.startSOPLangAgentSession(skillDescriptions, 'Run the plan', {
    planOnly: false,
});

Uniformity Across Skills

A single LLMAgent instance is shared throughout the system. This is deliberate: it ensures that all skills inherit the same configuration, rate limits, and logging behavior. Skills are not expected to create their own model clients or bypass the agent. That uniformity is what keeps orchestration behavior stable across subsystems.

Response Shapes and Validation

LLMAgent supports multiple response shapes to keep parsing decisions centralized. When the caller requests JSON, the agent parses and validates the output. When the caller requests code, the agent strips fences and returns executable text. This single location for parsing logic reduces fragile patterns in skills and keeps the system consistent.

Where It Fits

LLMAgent is the underlying engine for the entire library, but it is not the product-facing entry point. In most cases, users interact with RecursiveSkilledAgent, which then uses the LLMAgent internally. You still interact with LLMAgent directly when building custom skills or session workflows that require raw access to the model.
