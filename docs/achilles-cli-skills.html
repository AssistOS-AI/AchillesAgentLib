Achiles CLI — Skill Deep Dive

This companion guide explains how the CLI talks to onboard orchestrators, how plans are interpreted, and how the fallback generic-skill keeps requests productive.

Planning Pipeline

  1. Collect context — prompt history, global/user/session memories, workspaceRoot, and specsRoot are bundled into the plan request.
  2. LLM plan — the shared LLMAgent receives the high-level prompt and returns a JSON array of steps: { skill, prompt }.
  3. Validation — any skill not present in .AchillesSkills is removed (with a warning). If the filtered plan is empty, generic-skill is invoked automatically.
  4. Execution — each step runs through RecursiveSkilledAgent, inheriting the shared LLMAgent, prompt reader, and memory context.
  5. Memory routing — after the plan finishes (or pauses) the CLI summarizes the outcome and asks the LLM whether the interaction should be saved to global, user, or session memory.

Bootstrap Skills

Bootstrapping is only triggered when .specs/ does not exist:

  • ignore-files — initializes .specs/.ignore with default patterns. Users can rerun the skill manually to add more folders later.
  • reverse-specs — reverse engineers the workspace into DS documents. The resulting DS files feed build-code and refactor-design.

Operational Skills

Skill Purpose Inputs/Outputs
update-specs Creates/updates URS, FS, DS, and test stubs based on LLM directives. LLM plan actions (createURS, createFS, createDS, describeFile, createTest). Writes to .specs/.
build-code Reads DS file-impact sections and rewrites only the referenced files. Skips files outside the DS manifests; prepends “Managed by DS-###” banners.
mock-build Generates HTML or CLI previews under .specs/mock. Auto-selects output type based on dependencies (react, etc.).
run-tests Runs runAlltests.js (or suite-specific scripts) and captures stdout/stderr. Returns exit codes, counts, and trimmed logs so agents can react.
fix-tests-and-code Wraps run-tests in a loop, capturing failures in DS documents and invoking build-code. Stops after success or when attempts are exhausted; DS records the history.
generate-docs Outputs HTML snapshots plus an index.html that links to every URS/FS/NFS/DS. Each DS HTML uses the friendly filename (e.g., DS-003-fix-loop-ds.html).
refactor-design Captures refactor prompts as DS updates and automatically builds referenced files. Generates DS slugs (e.g., DS-004-refactor-runner) so multiple designs are easy to find.

generic-skill

The fallback orchestrator exposes a deterministic tool API:

  • list-files — lists directory contents (excludes dotfiles) up to 50 entries.
  • read-file — prints the first ~8 KB with a byte count.
  • rewrite-file — packages the current content and the operator’s prompt, then asks the LLM to emit a complete replacement.
  • replace-text — deterministic search/replace (no LLM).
  • create-file, append-file, delete-path — filesystem helpers.

generic-skill is ideal for lightweight maintenance (tweaking README, generating a script, etc.) or as a safety net when the planner cannot identify a specialized orchestrator.

Shared Context & Memory

Every orchestrator receives the following keys in context:

  • workspaceRoot & specsRoot
  • the shared llmAgent
  • logger (prints [skill:name] ... lines during execution)
  • globalMemory, userMemory, sessionMemory snapshots

Skills may ignore the memories, but they are invaluable when composing multi-step plans (e.g., spec-mentor can include past operator preferences).
