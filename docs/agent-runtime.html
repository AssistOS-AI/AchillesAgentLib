Agent Runtime

LLM mediation, recursive execution, and lifecycle APIs that keep Achilles Agents plans on track.

LLMAgent

LLMAgent centralises all communication with language models. It accepts an invoker strategy, which may route requests to OpenAI, Anthropic, a test stub, or any custom provider.

Constructor

new LLMAgent({
    name: 'PlanningLLM',
    invokerStrategy: async ({ prompt, context, history, mode }) => {
        // Dispatch to provider of your choice.
        return provider.invoke(prompt, { context, history, mode });
    },
});

Key Methods

  • complete({ prompt, history = [], context = {}, mode = 'fast' }): Returns the raw model output string.
  • interpretMessage(message, { intents }): Classifies user replies into accept, cancel, update, or ideas, using heuristics first and falling back to the model.
  • parseMarkdownKeyValues(markdown): Converts LLM output into key-value maps (used for argument extraction).
  • classifyMessage(message): Lightweight intent classifier when a full completion is not needed.

The agent enforces JSON parsing where required, logs debug traces when LLMAgentClient_DEBUG=true, and supports request cancellation by cancelling the underlying invoker promise.

LLMAgent Sessions

Beyond single-call completions, LLMAgent can drive longer-lived agentic sessions that maintain context across multiple prompts while calling tools repeatedly.

  • startAgentSession(tools, initialPrompt, options): creates an AgenticSession that orchestrates arbitrary tools. The session object exposes newPrompt(prompt) to continue the conversation on the same context, and getVariables() to inspect the final values or summary state.
  • startSOPSession(skillsDescription, initialPrompt, options): creates a SOPAgenticSession that uses LightSOPLang as the internal planning language. Each prompt refines or extends the underlying SOP script instead of starting from scratch, and the session exposes getPlan() to retrieve the current LightSOPLang source.

These session APIs are useful when you want the LLM to iteratively choose tools, revise plans, and accumulate state rather than recomputing everything from a single natural language prompt.

RecursiveSkilledAgent

RecursiveSkilledAgent supervises complex, multi-step plans. It can execute skills directly, run orchestration scripts, or call itself recursively when LightSOPLang commands reference other skills.

Constructor

const recursiveAgent = new RecursiveSkilledAgent({
    skilledAgent,   // host-provided skill runner
    startDir,       // directory containing orchestration fixtures
    reviewMode: 'none' | 'summary' | 'full', // optional
});

Execution APIs

  • executePrompt(prompt, options): Primary entry point. options may include:
    • skillName to force a specific skill.
    • availableTools for MCP contexts.
    • reviewMode to override global review behaviour.
  • executeWithReviewMode(prompt, options, reviewMode): Same as above but with explicit review mode.
  • cancel(): Propagates cancellation down the active stack.

Execution Flow

  1. Determines the target skill or LightSOPLang script.
  2. Loads orchestration metadata (allowed skills/tools, fallback behaviour, scripts).
  3. Executes LightSOPLang via registries that enforce availability filters.
  4. Delegates skill actions back to the hosting skill runner, collecting transcripts and outcomes.
  5. Aggregates plan entries, execution results, and fallback outcomes into a coherent response.

Review Modes

  • none: Execute immediately without review interrupts.
  • summary: Provide summarized outputs suitable for human approval.
  • full: Retain granular transcripts for auditing or human-in-the-loop workflows.

Error Handling & Cancellation

Both LLMAgent and RecursiveSkilledAgent propagate structured errors. For orchestration flows:

  • JSON parse failures fall back to deterministic LightSOPLang scripts (when available).
  • Unavailable skills or tools are marked as skipped, with reasons recorded in the plan.
  • User cancellation (interpreted via interpretMessage) bubbles up as Error('Skill execution cancelled by user.').
  • Auto-cancel heuristics (e.g., maintenance windows) yield STATUS_CANCELED values with machine-readable reasons.
