Runtime Behavior of Dynamic Code Generation Skills

This chapter explains the two execution paths of the subsystem and clarifies how temporary code execution differs from module-based execution.

Default Dynamic Executor

When no local module matching the skill folder name is present, the subsystem uses its default executor. In that path, the runtime builds a decision prompt from the descriptor guidance and from the current input. It asks the LLM for a JSON object that selects either text mode or code mode. If the result is text mode, the returned text becomes the skill output. If the result is code mode, the returned code snippet is executed immediately.

The temporary code execution path is intentionally ephemeral. The subsystem wraps the generated snippet in an async function and executes it through eval. If the snippet returns undefined, the runtime performs one fallback attempt by detecting the last declared function name and calling it automatically. If the result remains undefined, execution fails with an explicit error asking for a final return value.

Important

In the default path, generated code is not written to disk. It is executed for the current request only and then discarded.

Module Executor

When the skill directory contains a file whose name matches the folder name and whose extension is .mjs or .js, the subsystem prepares a module executor instead. In that case, the file is imported dynamically and the runtime looks first for an exported action function, then for a default function export. If no callable export is found, execution fails.

The payload passed to that module includes input, promptText, llmAgent, recursiveAgent, sessionMemory, context, user, attachments, the normalized llmMode, the current tier configuration, and the skill name. This means that module-based DCG skills still operate inside the shared agent runtime even though their local logic is implemented directly in code.

Input and Result Semantics

The operative input field for this subsystem is always input. If the caller supplies options.args.input, the subsystem uses that string. Otherwise it falls back to the user prompt text. An empty input is rejected before either executor runs.

The result semantics are similar across the two paths. String results are returned as they are. Null or undefined values become an empty string. Other values are serialized to JSON when possible and converted to string otherwise. In the default executor, the subsystem also attempts to append the user input and the produced output to session memory when such memory is available. In the module executor, that persistence depends on the module implementation itself.
