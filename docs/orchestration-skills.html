Orchestration Skills

Skills that define the flow of complex work without executing it directly.

Vision

Orchestration Skills exist to offload complex, multi-step coordination from the RecursiveSkilledAgent into explicit, reusable workflows. They do not perform execution themselves. Instead, they describe how to break a task into smaller skills and how those skills should be invoked to produce a complete result.

This makes complex work repeatable. A procedure such as producing software documentation is too large for a single skill, but it becomes tractable when an orchestration skill defines the flow and delegates to focused skills that each do one thing well.

How They Work

When an orchestration skill is selected, its subsystem launches an agentic session that plans and coordinates the smaller skills. The orchestration skill defines the rules of the flow, while the session decides which step to execute next until the overall request is satisfied. This keeps the RecursiveSkilledAgent lean while enabling sophisticated workflows.

The two agentic session styles are introduced on the Agentic Sessions page. Orchestration skills can trigger either style depending on their configuration.

Agentic Sessions Used by Orchestration Skills

Orchestration skills can launch one of two session types. The choice defines how planning and execution are carried out.

  • Loop sessions are ReAct-style iterations where the session chooses a skill, executes it, and repeats until done.
  • Soplang sessions generate a plan first and then execute it deterministically.

The session type is controlled by the Session section in the orchestration skill file. If it is set to loop, the Loop session is used; otherwise the Soplang session is the default.

Oskill Sections

Orchestration skills are defined in oskill.md. The sections below control how the orchestrator behaves and which skills it is allowed to call.

  • Instructions defines the system guidance for the orchestrator.
  • Description states the purpose and expected outcome.
  • Allowed-Skills restricts which skills can be used during execution.
  • Allowed-Prep-Skills optionally restricts which skills can run during preparation.
  • Preparation defines a pre-execution context build step for Soplang sessions.
  • Session chooses between loop or default Soplang behavior.

Preparation Phase

Soplang sessions need a stable context before the plan is generated and executed, because the plan itself cannot absorb new context on the fly. The Preparation section exists to solve this. It allows a short, controlled phase that gathers or synthesizes context before the plan is produced.

Preparation uses a restricted set of skills defined by Allowed-Prep-Skills, so the context-building step remains bounded and predictable. The result is a cleaner, more deterministic Soplang execution phase.

Folder Structure

Orchestration skills live in their own folder, with a single oskill.md descriptor file. 

skills/
└── documentation-orchestrator/
    └── oskill.md

Example oskill.md

The example below shows a typical orchestration skill that uses a preparation phase and a restricted allowlist to coordinate other skills. 

# review-specs

## Intents
The review-specs skill analyzes existing specification files for gaps,
inconsistencies, errors, and areas needing improvement. It examines all
markdown files under the project's specs directory, records findings as
issues and proposed fixes in the specs backlog.

## Preparation

1. Load files that are mentioned in the user request. If none are mentioned,
   load all spec files under `./docs/specs` (only .md files) using
   context-loader.

- Note: always load `./AGENTS.md`.

## Allowed Preparation Skills
- context-loader

## Instructions

1. Determine internally which files have problems.
2. For each file that has problems, call backlog-expert once with the file
   content and the detected problem to generate task descriptions.
3. Add all generated tasks to the specs backlog using backlog-api
   addTasksFromText.
4. For each added task ID, call backlog-expert again with the task description
   plus only the relevant spec file contents (planner decides which files are
   relevant) to generate numbered options (1., 2., 3. only, no extra prose).
5. Add the generated options to the task using backlog-api addOptionsFromText.

## Allowed Skills
- file-system
- backlog-api
- backlog-expert
