Agentic Sessions

The execution core that holds context, chooses skills, and drives multi-step reasoning.

Vision

Agentic sessions are the “brain” of the system. They are the mechanism by which an agent keeps context, selects skills, and iterates until a user request is satisfied. They are not an optional layer; they are where the system actually reasons about what to do next.

This is why skills exist at all. A session chooses which skill to call, records the result, and uses that result to decide the next step. That loop is the core execution model for both adaptive and plan‑driven workflows.

Sessions are in‑memory only. They do not persist conversation state to disk; they keep a working history for the duration of the execution and then end.

Two Session Types

The system exposes two agentic session types, each with a different tradeoff between flexibility and cost. Both use a skill allowlist provided by the caller and both rely on the same reserved completion skills.

  • Loop session: dynamic, ReAct-style reasoning and execution.
  • Soplang session: plan-first execution using a deterministic script.

Loop Session

A Loop agentic session receives a list of skills and starts by choosing which skill is most likely to move the user request forward. It executes that skill, records the result in context, and then repeats the decision for the next step. This continues until the request is satisfied.

The Loop session is implemented in LLMAgents/AgenticSession.mjs as LoopAgentSession. Each step requires a new LLM decision, which makes the session highly adaptive but also more expensive than plan‑first execution.

Loop sessions are used by orchestration skills when the descriptor explicitly requests a loop session, and they are also the session type used by the Anthropic Skills subsystem.

Usage

Use Loop sessions when the workflow is uncertain, when intermediate results can change the path forward, or when you need to handle user confirmation and interactive steps without pre‑computing a plan. In this system they are selected by orchestration skills that opt into loop behavior and by the Anthropic Skills subsystem.

A Loop session is organized into turns. A turn represents a single user prompt and the chain of skill decisions that follow it. During a turn, the session can execute multiple skill calls, record intermediate results, and decide when the request has been satisfied. When the session ends, the turn is closed and its history remains available for the rest of the session.

Loop sessions include two reserved built‑in skills: final_answer and cannot_complete. The LLM can call final_answer when it determines the goal is met, or cannot_complete when it determines the request cannot be satisfied. Either call ends the session.

Results from each skill call are stored as variables. When a future skill needs a previous result, the LLM can reference it using $$variable, which is replaced with the actual value at execution time. This avoids re‑typing large outputs and reduces hallucination risk.

If a skill returns requiresConfirmation or requiresInput, the session pauses in an awaiting‑input state and waits for the user to respond. The next prompt is interpreted as a continuation unless the user clearly signals a new task.

The loop has safeguards that prevent repeated execution of the same skill with the same parameters. Errors are recorded in the session history and a failed call cannot be retried with the identical invocation.

The session also enforces execution limits such as maximum steps per turn and maximum errors. When these limits are reached, the session ends with a failure response rather than looping indefinitely.

Perks

Loop sessions are more adaptive than generic agent loops because they operate over a strict skill allowlist and persist each skill result as a variable that can be referenced safely. This reduces hallucination risk when outputs are large or structured. They also support interactive pauses when a skill requires confirmation or input, which is crucial for operational workflows.

The tradeoff is cost and latency. Every step requires another LLM decision, so a long chain of skills results in multiple model calls. That is acceptable when the workflow is dynamic, but it can be inefficient for predictable tasks where a plan can be generated up front.

Soplang Session

A Soplang session is plan‑first. It receives the allowed skills and asks the LLM to generate a Soplang plan that describes the complete execution. Each command in the plan represents a skill invocation; results are stored in variables and referenced later using $varName.

The Soplang session is implemented in LLMAgents/SOPAgenticSession.mjs as SOPAgenticSession. Planning is performed once per attempt, and the resulting script is executed by the LightSOPLang interpreter.

Soplang sessions are the default execution mode for orchestration skills. They are selected when the orchestration descriptor does not request a loop session.

Usage

Use Soplang sessions when the workflow is stable and repeatable, and when you want predictable cost and latency. In this system they are the default for orchestration skills and are used whenever a skill descriptor does not explicitly request Loop behavior.

Once the plan is generated, it is executed deterministically. If execution succeeds, the session completes. The plan must invoke final_answer to finish successfully, or cannot_complete if the request cannot be satisfied.

If execution fails, the session can replan a limited number of times. The captured errors are injected into the next planning attempt so the LLM can repair the plan and retry.

Soplang sessions can also pause when a skill requires confirmation or input. In that case the session records the pending skill and resumes with a direct plan that targets that skill once the user responds.

Perks

Soplang sessions are more cost‑predictable than generic agent loops because the plan is generated once and then executed without repeated LLM decisions. The plan itself is explicit and inspectable, which makes it easier to audit and test. This matches the system’s focus on repeatable, skill‑bound workflows.

The tradeoff is rigidity. Because the plan is fixed up front, it cannot handle true branching or iteration beyond what was explicitly encoded. If the workflow needs to adapt to new information while running, a Loop session is usually the better fit.

Reserved Completion Skills

Both session types rely on two built‑in skills that mark the end of execution.

  • final_answer signals that the request has been fully satisfied.
  • cannot_complete signals that the request cannot be satisfied and includes a reason.
