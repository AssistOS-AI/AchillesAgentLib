Mirror Code Generator

The mirror code generator is the shared, internal utility that transforms specs/ markdown files into executable modules for code skills. It performs regeneration only when specs are newer than their target outputs and validates outputs through test generation, execution, and targeted repairs. When code is already up-to-date but tests are missing, it generates tests only and stops without updating backups.

Internal Skill: fds-generator

Generates specs/ FDS files from DS inputs (DS*.md). Mirror-code-generator checks for DS files only in the target root, docs/, and docs/specs/ (non-recursive). The FDS generator reads each DS file's Affected Files section. Each listed FDS path is resolved relative to the target directory and written as a .mjs.md spec file. Regeneration is forward-only: if a DS file is newer than its listed FDS (or the FDS is missing), the FDS is regenerated.

Internal Skill: tests-generator

Builds test plans and test files from the current on-disk codebase. It writes tests under tests/, creates tests/runAll.mjs when missing, and emits a mapping of test files back to their source modules. It does not execute tests; execution is handled by the mirror-code-generator workflow.

Internal Skill: mirror-code-generator

Converts specs/ markdown into executable modules under src/, runs syntax checks, generates tests via tests-generator, executes tests, and performs a single repair pass when failures occur. It updates specs/.backup after successful regeneration.

Generation Flow

The full generation pipeline combines the three internal skills. It checks for DS files and only runs the DS-to-FDS step when DS inputs exist and are newer than their listed outputs (or when specs/ is missing). The pipeline stops only when specs/ is still missing after that conditional step.

  1. Check for DS files and determine if FDS regeneration is required.
  2. Run fds-generator only when DS files exist and are newer than their listed outputs (or when specs/ is missing).
  3. If specs/ is still missing after the step above (no DS and no specs), stop.
  4. For each stale spec, generate code (spec + backup spec + existing code), write to src/, and syntax-check.
  5. If syntax check fails, repair once; on repeat failure, revert and skip tests for that file.
  6. Generate test plans and test files via tests-generator.
  7. Run tests/runAll.mjs; if failures occur, repair referenced files and re-run once.
  8. Replace specs/.backup after successful regeneration (skip when only tests were generated).
