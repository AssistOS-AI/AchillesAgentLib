Mirror Code Generator

The mirror code generator is the shared, internal utility that transforms specs/ markdown files into executable modules for code skills. It performs regeneration only when specs are newer than their target outputs and validates outputs through test generation, execution, and targeted repairs. When code is already up-to-date but tests are missing, it generates tests only and stops without updating backups.

Regeneration Preconditions

The generator runs only when a skill directory contains a specs/ folder with at least one .md or .mds file. Each spec file maps to a target output path under src/ by removing the specs/ prefix and the .md / .mds extension. A file is regenerated when the spec modification time is newer than the corresponding output file, or the output is missing. 

specs/index.mjs.mds  ->  src/index.mjs
specs/utils/foo.mjs.md  ->  src/utils/foo.mjs

Per-File Generation Workflow

For each spec file that requires regeneration, the generator performs a single-file LLM generation step, then validates the resulting code with a single test-generation prompt. If the spec includes a #Validation or #Testing section, the generator uses that text to derive test cases; otherwise it uses a default prompt focused on core functionality.

If no files need regeneration but tests/ is missing, the generator skips code generation and produces tests for the existing code. In that path, backup specs are not used for test prompts and no specs/.backup update is performed.

LLM Inputs for Code Generation

The code generation prompt is built from three sources:

  • Current spec content (the authoritative requirements).
  • Backup spec content from specs/.backup, when present (for diff-aware updates).
  • Existing code file content, when present (for incremental changes).

Testing and Repair Flow

The generator writes the generated code to the skill directory, runs a syntax check with node --check, then generates both positive test cases and a matching test runner for the module's public API (not necessarily an action export). The runner must always emit process.stdout.write(JSON.stringify({ results }));. The generator validates the runner with node --check and validates the test cases JSON via a Node parse. If either validation fails, it logs the error and skips testing (files remain written). Syntax check failures trigger a targeted repair pass; a second failure reverts to the previous on-disk version and skips tests for that file. Test failures trigger a targeted repair pass. The repaired code is re-tested once; remaining failures only emit warnings and do not block writing outputs. For test-only generation (no regeneration), failures are logged and the existing code is not modified.

  1. Generate code with the LLM (inputs listed above).
  2. Write generated code to src/ in the skill directory.
  3. Run node --check on the written file.
    • If it fails, repair using spec + backup + generated code + syntax error details.
    • If it fails again, revert to the previous on-disk version (or remove the file if none) and skip tests.
  4. Generate positive tests and a runner that matches the module's API.
    • Uses Testing/Validation section content if present; otherwise a default core-functionality prompt.
  5. Write tests/test-cases.json and tests/run-tests.mjs without validation.
  6. Validate the runner and test cases:
    • node --check on tests/run-tests.mjs
    • node -e "JSON.parse(require('fs').readFileSync(process.argv[1],'utf8'))" on tests/test-cases.json
    If either fails, log and skip testing (files remain written).
  7. If validations pass, execute tests from tests/.
  8. If tests fail, repair using:
    • Current spec content.
    • Backup spec content (when present).
    • Generated code (not the on-disk code).
    • Failure details (name, input, expectedOutput, actual).
    • Test cases and the generated runner (specs remain authoritative).
    If tests were generated without regeneration, no repair occurs; failures are logged.

Output and Backup Behavior

Generated output files are written under src/ in the skill directory, mirroring the specs/ tree. After all files are processed, the generator replaces specs/.backup with the current specs/ directory contents. When only tests are generated (no regeneration), backups are not updated.
