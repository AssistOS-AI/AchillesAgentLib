Mirror Code Generator

The mirror code generator is the shared, internal utility that transforms specs/ markdown files into executable modules for code skills. It performs regeneration only when specs are newer than their target outputs and validates outputs through test generation, execution, and targeted repairs. When code is already up-to-date but tests are missing, it generates tests only and stops without updating backups.

Internal Skill: mirror-code-generator

The mirror-code-generator is the orchestration layer that turns spec markdown into runnable code, validates the output, and repairs failures. It evaluates each spec file individually and only regenerates code when the spec is newer than its target output (or the output is missing). Each spec path maps to a target file under src/ by removing the specs/ prefix and the .md/.mds extension. 

specs/index.mjs.mds  ->  src/index.mjs
specs/utils/foo.mjs.md  ->  src/utils/foo.mjs

For every stale spec, the LLM receives three inputs: the current spec content, the previous spec from specs/.backup (when present), and the existing code file (when present). The generator writes the new code to disk, runs node --check, and triggers a repair pass on syntax failure. If a second syntax check fails, the previous on-disk version is restored (or the file is removed) and tests are skipped for that file.

After code is written, mirror-code-generator invokes tests-generator to build a test plan and create executable tests from the current on-disk codebase (no specs passed to test prompts). It then runs tests/runAll.mjs and performs a single repair loop across the referenced source files when failures occur, followed by one re-run. Remaining failures are logged but code is left as-is.

When no spec needs regeneration but tests/ is missing, it generates tests only and stops without updating specs/.backup. After successful regeneration, it overwrites specs/.backup with the current specs snapshot.

Internal Skill: tests-generator

The tests-generator builds a structured test plan from the current on-disk codebase and materializes it as executable test files. Plans can span multiple source files when behavior crosses module boundaries, and may declare fixtures when code reads from disk. Fixtures are written as UTF-8 text files under tests/ (including nested fixture directories when needed).

It ensures tests/runAll.mjs exists (copied from a template when missing) and returns a mapping from generated test files back to their source modules, which mirror-code-generator uses for targeted repair. Tests are generated in a format that writes JSON results to stdout, and individual test files are syntax checked on creation. This skill does not execute tests; execution and repair are handled by mirror-code-generator.

Internal Skill: fds-generator

The fds-generator materializes specs/ files from DS inputs (DS*.md). Mirror-code-generator searches for DS files only in the target root, docs/, and docs/specs/ (non-recursive). Each DS file must contain an Affected Files section; listed entries are resolved relative to the target skill directory and written as .mjs.md spec files.

Regeneration is forward-only: when a DS file is newer than one of its listed FDS outputs (or the output is missing), the corresponding spec is re-created. If specs are missing entirely and DS files exist, the fds-generator runs to repopulate specs/ before mirror-code-generator proceeds.

Start
  |
  v
Check for DS files?
  |-- No --> Skip fds-generator
  |
  v
DS files found -> specs missing or any FDS older than DS?
  |-- No --> Skip fds-generator
  |-- Yes --> Run fds-generator
  |
  v
Check specs/ exists?
  |-- No --> Stop (no DS and no specs)
  |
  v
Check if any spec is newer than its code?
  |-- No --> Tests missing?
  |           |-- Yes --> Generate tests only (no backup spec, no backup update) --> Done
  |           |-- No  --> Stop
  |
  v
For each spec in specs/ (excluding .backup)
  |
  v
Read spec -> Gather spec + backup + existing code
  |
  v
Generate code -> Write to src/
  |
  v
Syntax check (node --check)
  |-- Fail --> Repair (spec + backup + generated code + syntax error)
  |           -> Rewrite -> Recheck
  |           -> Fail? Revert + skip tests
  |-- Pass --> Continue
  |
  v
Build test plan from current codebase
Generate test files + optional fixtures
Write tests/ (create folders as needed) and fixtures
Syntax-check each test file
Ensure tests/runAll.mjs (copy template if missing)
Run tests/runAll.mjs (executes all .mjs/.js under tests/)
Failures? -> Repair iteratively across referenced source files
          -> Rerun once
          -> Warn if still failing
  |
  v
Overwrite specs/.backup
  |
  v
Done
