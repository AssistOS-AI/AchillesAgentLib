Mirror Code Generator

The mirror code generator is the shared, internal utility that transforms specs/ markdown files into executable modules for code skills. It performs regeneration only when specs are newer than their target outputs and validates outputs through test generation, execution, and targeted repairs. When code is already up-to-date but tests are missing, it generates tests only and stops without updating backups.

Regeneration Preconditions

The generator runs only when a skill directory contains a specs/ folder with at least one .md or .mds file. Each spec file maps to a target output path under src/ by removing the specs/ prefix and the .md / .mds extension. A file is regenerated when the spec modification time is newer than the corresponding output file, or the output is missing. 

specs/index.mjs.mds  ->  src/index.mjs
specs/utils/foo.mjs.md  ->  src/utils/foo.mjs

Per-File Generation Workflow

For each spec file that requires regeneration, the generator performs a single-file LLM generation step, then validates the resulting code through a two-step testing workflow. The testing flow now analyzes the generated codebase to produce a structured test plan, then generates concrete test files, optional test-case JSON artifacts, and optional persistent fixtures under tests/ based on that plan. Test plans can span multiple source files when behavior crosses module boundaries. When code reads from disk, the plan should describe the fixture layout.

If no files need regeneration but tests/ is missing, the generator skips code generation and produces tests for the existing code. In that path, backup specs are not used for test prompts and no specs/.backup update is performed.

LLM Inputs for Code Generation

The code generation prompt is built from three sources:

  • Current spec content (the authoritative requirements).
  • Backup spec content from specs/.backup, when present (for diff-aware updates).
  • Existing code file content, when present (for incremental changes).

Testing and Repair Flow

The generator writes the generated code to the skill directory, runs a syntax check with node --check, then generates test plans and test files using the current on-disk code. Each test file is responsible for emitting process.stdout.write(JSON.stringify({ results })); where results contains objects with expected, actual, and pass. Test generation can also return a fixtures array of files to write under tests/ (or a nested fixture directory). A shared tests/runAll.mjs runner discovers and runs every .mjs/.js test file under tests/ recursively, aggregates failures, adds the originating test file name, and returns a unified JSON summary under failedTests. Test files are syntax-checked individually; failures emit warnings but do not block the run. When tests fail after regeneration, the generator attempts a single repair pass across all source files referenced by the failing test plan, then re-runs the tests once. For test-only generation (no regeneration), failures are logged and the existing code is not modified.

  1. Generate code with the LLM (inputs listed above).
  2. Write generated code to src/ in the skill directory.
  3. Run node --check on the written file.
    • If it fails, repair using spec + backup + generated code + syntax error details.
    • If it fails again, revert to the previous on-disk version (or remove the file if none) and skip tests.
  4. Generate a structured test plan from the current codebase (no specs provided).
  5. Generate one or more executable test files plus optional JSON test-cases and fixtures.
  6. Write test files and fixtures under tests/ (create folders as needed) and syntax-check each test file.
  7. Ensure tests/runAll.mjs exists (copied from the template when missing).
  8. Execute tests/runAll.mjs to run all tests and collect failedTests.
  9. If tests fail after regeneration, repair iteratively across all referenced source files, then re-run once.
  10. Log remaining failures without additional repair.

Output and Backup Behavior

Generated output files are written under src/ in the skill directory, mirroring the specs/ tree. After all files are processed, the generator replaces specs/.backup with the current specs/ directory contents. When only tests are generated (no regeneration), backups are not updated.
