Overview

ploinkyAgentLib is a production-grade framework for orchestrating conversational agents that execute discrete skills while collaborating with large language models (LLMs). It blends deterministic execution flows with AI-assisted argument collection, semantic search, and scripted orchestration via LightSOPLang.

  • ES module library (.mjs) with modern JavaScript conventions.
  • Supports semantic skill ranking, AI-guided selection, and interactive argument gathering.
  • Integrates with Model Context Protocol (MCP) tools and LightSOPLang scripts.
  • Includes an extensive test harness mirroring real conversational scenarios.

Architecture

The library is organised around a central SkilledAgent with supporting subsystems:

Layer Responsibility Key Modules
Presentation Orchestrators, LightSOPLang scripts, MCP bridges OrchestratorSkillsSubsystem, MCPSkillsSubsystem
Execution Interactive argument collection and validation SkilledAgents/executor/context.mjs, mainLoop.mjs
Registry Skill definitions, search index, argument metadata SkilledAgents/SkillRegistry.mjs
LLM Abstraction Prompt orchestration, completion parsing, heuristics LLMAgents/LLMAgent.mjs, SkilledAgents/executor/llm.mjs

All modules observe strict 4-space indentation, single quotes, trailing commas on multiline literals, and semicolons to prevent ASI ambiguity.

Core Components

SkilledAgent

The facade exposed to hosting applications. It wires an LLMAgent and a SkillRegistry, and routes execution requests through the interactive main loop. 

import { SkilledAgent } from 'ploinkyAgentLib/SkilledAgents';

const agent = new SkilledAgent({ llmAgent });
agent.registerSkill({...});
const result = await agent.executeSkill('add-equipment', { taskDescription });

Skill Registry

  • Normalises skill spec objects and validates required metadata.
  • Extracts enumerators, validators, resolvers, and presenters.
  • Builds FlexSearch indexes for semantic ranking.
  • Supports argument aliases and rich options with synonyms.

Execution Context

Created per invocation and responsible for applying user-provided arguments, enforcing types, placeholder detection, and enumerator resolution. JSON export uses canonical argument names only.

Execution Lifecycle

  1. Skill discovery: Task text is ranked via FlexSearch; LLM may confirm the best match.
  2. Context creation: Enumerators load options, aliases map to canonical keys, resolvers and validators attach.
  3. Argument loop: Missing arguments trigger user prompts. Inputs flow through LLM extraction and manual parsing.
  4. Confirmation: If needConfirmation is not false, a natural language summary is produced; user intent is classified via heuristics and LLM fallback.
  5. Execution: Validated arguments feed the registered action. Outcomes propagate through CLI/reporting layers.

Placeholders like “your_job_name” or “not provided” are detected and rejected unless they are legitimate values. Enumerated arguments support fuzzy matching, synonyms, and tolerant Levenshtein distance heuristics.

LightSOPLang Interpreter

LightSOPLang provides repeatable, auditable orchestration scripts. Each variable resolves by executing commands via a registry implementing executeCommand(payload, responder).

  • Payload: includes the command name, resolved arguments, raw text, and variable metadata.
  • Responder API: success, fail, and cancel helpers produce typed values.
  • Auto cancel: Heuristic detection funnels through normalizeCancelReason to de-duplicate reasons.
  • Execution monitor: Enforces command limits and cancellation heuristics per interpreter instance.

Recent updates added script-aware registries to orchestrators and MCP skills, allowing LightSOPLang sections in skill descriptors to define canonical execution flows without relying solely on LLM plans.

Model Context Protocol (MCP) Integration

The library natively integrates with MCP servers, enabling agent-driven tool invocations. Key features:

  • Allowed tools: Skills whitelist tool identifiers, ensuring prompts cannot call unapproved MCP commands.
  • Scripted plans: LightSOPLang scripts can schedule MCP tool usage deterministically.
  • Dynamic plans: LLM-driven plans fall back to deterministic scripts when JSON responses cannot be parsed.

Testing Strategy

  • Node’s built-in test runner powers suite files under tests/.
  • Use node tests/runAllTests.js (aliased via npm test) to run the full matrix.
  • Individual suites target scenarios such as interactive skill flows, LightSOPLang execution, and recursive agent behaviour.
  • Helpers under tests/useSkill/helpers.mjs provide instrumentation for scripted conversations and LLM interception.

When altering orchestrator subsystems or executor logic, the entire suite must pass. Specific test files that must remain green include:

  • SkilledAgents/tests/*.mjs
  • tests/useSkill/*.mjs
  • tests/recursiveAgent/orchestration.test.mjs
  • tests/lightSOPLang/*.mjs

Development Checklist

Before submitting contributions, ensure the following:

  1. Read and understand SKILLEDAGENT-ARCHITECTURE.md.
  2. Use ES modules (import/export), 4-space indentation, trailing commas, single quotes, and semicolons.
  3. Document public APIs with JSDoc and provide descriptive error messages.
  4. Write or update tests alongside every feature or bug fix.
  5. Run npm test (or targeted suites) before pushing changes.
  6. Keep commits focused, and prefer present-tense imperative messages.
  7. Avoid destructive Git commands that could override the user’s worktree.
