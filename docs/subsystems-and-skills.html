Subsystems and Skills

Why skills are categorized, and how subsystems enforce consistent behavior for each category.

Vision

Skills in this library are not all the same. They represent different tradeoffs between flexibility and control, so we classify them into categories with shared expectations. That classification is what makes behavior predictable across a large skill catalog. Subsystems exist to encode those expectations so each skill is handled in the correct way.

A dynamic skill that generates code on the fly has very different safety and execution needs than a skill that executes stable, pre-generated logic. Treating them the same would either add too many constraints to flexible skills or too much risk to stable skills. Subsystems resolve that tension by giving each category its own handling path.

Skills vs Tools

A tool is typically a static, deterministic function. A skill is intentionally more capable: it receives access to the shared LLMAgent, can reason with its own context, and can choose how to respond within the guardrails defined in its skill file. This adds a controlled level of non-determinism that helps the system handle real-world variability without becoming a generic agent.

In practice, a skill is a compromise between a fully open-ended agent and fixed code. It is specialized around a task, but it retains just enough flexibility to cope with imperfect inputs and evolving requirements.

What Subsystems Do

A subsystem is the runtime that knows how to interpret a skill file, prepare it, and execute it safely. When a skill is invoked, the RecursiveSkilledAgent delegates execution to the subsystem that matches the skill’s type. That subsystem applies the right preparation rules, execution flow, and safety checks for that category.

This design keeps the overall system stable. The agent does not need to understand every nuance of every skill type because each subsystem encapsulates that complexity.

Skill Registration Requirements

A skill is only registered if it has a descriptor file in Markdown. Each skill type has its own descriptor file name and its own set of sections, but the pattern is consistent: a skill is a structured document first, and an executable component second. This is why skills are discoverable and why they can be reasoned about by the agent.

Skills that execute code must expose an action function as the entry point. The subsystem loads that function and calls it with a shared execution context.

Input Model and Shared Context

Every skill receives its primary input as a natural‑language promptText string, not as a strict JSON object. This is intentional. Skills are intelligent components, not static utilities, so they need room to interpret, clarify, and reason about imperfect inputs. The skill’s descriptor defines the expected structure, but the runtime always starts from plain language.

Each skill also receives the shared LLMAgent so it can interact with the model using the same configuration as the rest of the system. This makes skill execution feel like one agent handing off instructions to another specialized agent, rather than calling a rigid tool.

Skill Categories and Their Subsystems

Each skill type has its own subsystem with a distinct contract. The detailed behavior is documented on the dedicated pages for each category.

  • Orchestration Skills coordinate multi-step workflows and agentic sessions.
  • DBTable Skills model data tables and structured CRUD flows.
  • Code Skills execute stable, generated code from specifications.
  • Anthropic Skills integrate the Anthropic skill standard as static descriptors and guidance.
  • Dynamic Code Generation Skills generate and run code at runtime for flexible logic.
  • MCP Skills plan and optionally execute MCP toolchains as skill-driven workflows.

How Subsystems Shape Behavior

The differences between categories are not superficial. They affect how inputs are parsed, how execution is constrained, and how results are normalized. A Code Skill emphasizes reproducibility; a Dynamic Code Generation Skill emphasizes adaptability; an Orchestration Skill emphasizes planning and dependency management. Subsystems make those differences explicit and enforce them consistently across the library.

Folder Structure for Skills

Skills are organized by type and stored in dedicated folders. The layout below shows the common structure users encounter when building a skill catalog. 

skills/
├── my-orchestrator/
│   └── oskill.md
├── my-db-skill/
│   └── tskill.md
├── my-code-skill/
│   └── cskill.md
├── my-dynamic-code-skill/
│   ├── dcgskill.md
│   └── my-dynamic-code-skill.mjs
├── my-mcp-skill/
│   └── mskill.md
└── my-anthropic-skill/
    └── skill.md

The RecursiveSkilledAgent scans these directories, parses each skill file, and registers a skill record with its type. The appropriate subsystem is then responsible for preparing and executing that skill.
