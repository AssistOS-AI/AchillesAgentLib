Achilles CLI

Task-focused orchestration for the Achilles Agents Library. The CLI plans multi-step workflows, captures conversational memory, and allows operators to pause/resume expensive runs while keeping specifications in sync.

Looking for the full command reference or per-skill details? See Command & Prompt Guide and Skill Deep Dive.

Workspace Bootstrapping

The CLI only bootstraps when it discovers a fresh workspace (no .specs/ directory). During that first run it executes:

  1. ignore-files — seeds .specs/.ignore with defaults (e.g., node_modules, dist) plus anything the user has requested. Future reverse-spec scans use this list to skip heavy directories.
  2. reverse-specs — walks the workspace (respecting the ignore list) and records every source file inside DS documents. Those file-impact sections later drive build-code and mock-build.

Once .specs/ exists the CLI never re-runs the bootstrap step automatically. Developers can invoke the orchestrators manually whenever they need to refresh the ignore list or reverse-generated DS files.

Execution Flow

  1. User prompt → the CLI gathers global/user/session memory, then asks the LLM to plan the next actions.
  2. Plan filtering → only skills present in .AchillesSkills (local plus workspace roots) survive. If the plan is empty the CLI switches to generic-skill.
  3. Skill execution → each orchestrator receives workspaceRoot, specsRoot, the shared LLMAgent, the prompt history, and the up-to-date memory containers. Skills can therefore read or update both the file system and conversational context.
  4. Streaming & memory → as soon as a plan step completes the CLI prints a structured status block, then routes the outcome through an LLM-based memory classifier. Classified entries are stored in .specs/.history_global_memory, .history_user_memory, or the transient session memory.
  5. Pause/resume → users can hit ESC, Ctrl+C, or type /cancel to pause the current plan. The CLI remembers which step was running and later resumes via natural language (e.g., “continue”), continua, or /continue (optionally with extra instructions to replan).

This flow makes Achilles CLI usable directly by humans while remaining fully scriptable for other agents inside the library.

Built-in Orchestrator Skills

Skill Summary Key Notes
ignore-files Maintains .specs/.ignore so reverse-spec scans stay focused. Always injects defaults (node_modules, dist, caches) then merges user entries.
reverse-specs Describes every relevant source file in a DS file for traceability. Respects the ignore list and records exports/snippets for build-code.
update-specs Creates URS/FS/DS scaffolding for new requirements. Useful for ad-hoc requirement capture straight from a prompt.
mock-build Generates CLI or HTML mock artefacts under .specs/mock. Detects project type via package.json; ideal for rapid demos.
build-code Reads DS file-impact sections and rewrites stubs with managed banners. Only touches files referenced by DS metadata; prevents accidental edits.
generate-docs Exports HTML snapshots (.specs/html_docs) for URS/FS/NFS/DS. Use after significant specification changes for stakeholder review.
run-tests Executes runAlltests.js (optionally filtered by suite). Returns stdout/stderr plus exit codes for CI ingestion.
fix-tests-and-code Iteratively runs tests, records failures in a DS, and re-runs build-code. Capped at three attempts before reporting exhaustion.
refactor-design Captures refactor prompts as DS updates, annotates file impacts, and regenerates stubs. Accepts explicit DS-### references or creates new URS/FS/DS triplets.
generic-skill Fallback orchestrator that plans arbitrary file operations with LLM supervision. Tools include listing folders, reading files, LLM-driven rewrites, and targeted search/replace.

The Generic Skill

When no plan is produced (or a user explicitly requests it), generic-skill executes a small tool plan. The LLM receives the list of tools and produces JSON steps such as:

[
  {"tool":"list-files","target":"src"},
  {"tool":"read-file","target":"src/index.mjs"},
  {"tool":"rewrite-file","target":"src/index.mjs","instructions":"Add a default export"}
]

Each tool is sandboxed to the current workspace root. Rewrites pipe the existing file content back through the shared LLMAgent and expect a full file replacement wrapped in a code fence. For small edits, replace-text offers deterministic replacements without invoking the LLM. See the skill deep dive for every tool signature.

Prompting, Commands, and History

Achilles CLI keeps a rolling prompt history under .specs/.prompts_history. The readline buffer supports multi-line input (end a line with \ to continue) and arrow-key navigation. All runtime commands are summarized below — visit the command reference for details.

Command Purpose
/list Print the skill catalog (name, type, summary, implementation).
/help Show command hints, history shortcuts, and cancellation tips.
/debug [on|off] Toggle verbose LLM logging (LLM_DEBUG: request/response pairs).
/continue [<extra>] Resume the last plan (optionally replan with fresh instructions before resuming).
/cancel Pause the active plan, preserving its remaining steps for a later /continue.
/exit, /quit Close the CLI. The prompt history remains on disk for the next session.

At any time, pressing ESC or Ctrl+C sends the same cancellation signal. For more advanced usage (non-interactive mode, alternative skill roots, memory routing) see the command guide.

Language Enforcement & Spec Visibility

Regardless of the operator’s locale, the CLI enforces a language contract on every LLM call. The planner prompt and each skill execution include a “Language Requirements” block that defaults to English and rejects placeholder answers (for example “your_job_name”). Use /lang <language> to switch the contract; invoking /lang without arguments simply prints the current language so you can confirm the setting.

During plan execution the CLI streams spec activity back to the operator. When an orchestration skill emits createURS, createFS, createDS, describeFile, or createTest actions, the terminal prints a live [spec] preview and the final summary includes a concise spec actions: line listing the affected IDs and file paths. This makes it easy to see which requirements the LLM just modified without opening the Markdown files.

LLM Logs & Status Buckets

Every workspace keeps its own LLM telemetry under .specs/.llm_logs (per-call JSON lines) and .specs/.llm_stats (aggregated counters). The /status command renders:

  • Request counts and token totals per model.
  • Duration-based buckets (<100ms, <1000ms, <10000ms, <100000ms, >100000ms) populated whenever a bucket has at least one request.
  • The absolute paths to the log and stats files so you can tail them in another terminal.

Because the data sits inside the project’s .specs tree, switching workspaces automatically scopes logs to the correct context.

Testing the CLI

  • Temporary Workspaces — every test places its fixtures under tests/.tmp/<scenario>. Nothing writes to the repository root, and the directory is ignored by Git.
  • Per-test Timeouts — each node:test case explicitly sets a timeout (10–30 seconds) to prevent accidental hangs caused by blocking prompts.
  • Standalone Suites — CLI-specific tests live under tests/cli/gamp and exercise bootstrapping, plan execution, pause/resume, and the fallback pipeline.
  • Manual Runs — to reproduce an individual scenario, invoke node tests/cli/gamp/achilles-cli.test.mjs --test-name-pattern "scenario".
