Achiles CLI

Task-focused orchestration for the Achilles Agents Library. The CLI automatically plans, bootstraps specifications, and delegates work across orchestration skills—including a new generic-skill that can manipulate project files with LLM supervision.

Workspace Bootstrapping

Every CLI session starts inside the current workspace (or the explicit workspaceRoot argument when embedding). Before any user prompt is processed, the CLI runs two internal orchestrators:

  1. ignore-files — seeds .specs/.ignore with default Node.js artefacts and any user-selected folders. Keeping ignore entries local prevents accidental scans of node_modules, build artefacts, or large caches.
  2. reverse-specs — walks the workspace (respecting the ignore list) and describes each source file inside a dedicated DS document. The descriptions power build-code and keep the reverse-generated documentation up to date.

This bootstrap step is logged to the terminal (e.g., [auto] Running ignore-files-orchestrator – Automatic bootstrap…) and only runs once per CLI instance. Because each CLI test spins up a temporary workspace, bootstrapping never writes to the repository root.

Execution Flow

  1. User prompt → the CLI requests a JSON plan from the LLM that maps sub-tasks to orchestrator skills.
  2. Plan filtering → only skills present in .AchillesSkills are executed. Malformed or empty plans automatically fall back to generic-skill.
  3. Skill execution → each orchestrator receives common context (workspaceRoot, specsRoot, and the shared LLMAgent) so they can read or update the specification singleton (GampRSP).
  4. Fallback → if no plan was produced, the CLI invokes generic-skill to execute a lightweight tool-based plan using the same LLM.

All steps stream JSON-style status lines to the terminal so humans and automated agents can observe progress without diving into logs.

Built-in Orchestrator Skills

Skill Summary Key Notes
ignore-files Maintains .specs/.ignore so reverse-spec scans stay focused. Always injects defaults (node_modules, dist, caches) then merges user entries.
reverse-specs Describes every relevant source file in a DS file for traceability. Respects the ignore list and records exports/snippets for build-code.
update-specs Creates URS/FS/DS scaffolding for new requirements. Useful for ad-hoc requirement capture straight from a prompt.
mock-build Generates CLI or HTML mock artefacts under .specs/mock. Detects project type via package.json; ideal for rapid demos.
build-code Reads DS file-impact sections and rewrites stubs with managed banners. Only touches files referenced by DS metadata; prevents accidental edits.
generate-docs Exports HTML snapshots (.specs/html_docs) for URS/FS/NFS/DS. Use after significant specification changes for stakeholder review.
run-tests Executes runAlltests.js (optionally filtered by suite). Returns stdout/stderr plus exit codes for CI ingestion.
fix-tests-and-code Iteratively runs tests, records failures in a DS, and re-runs build-code. Capped at three attempts before reporting exhaustion.
refactor-design Captures refactor prompts as DS updates, annotates file impacts, and regenerates stubs. Accepts explicit DS-### references or creates new URS/FS/DS triplets.
generic-skill Fallback orchestrator that plans arbitrary file operations with LLM supervision. Tools include listing folders, reading files, LLM-driven rewrites, and targeted search/replace.

The Generic Skill

When no plan is produced (or a user explicitly requests it), generic-skill executes a small tool plan. The LLM receives the list of tools and produces JSON steps such as:

[
  {"tool":"list-files","target":"src"},
  {"tool":"read-file","target":"src/index.mjs"},
  {"tool":"rewrite-file","target":"src/index.mjs","instructions":"Add a default export"}
]

Each tool is sandboxed to the current workspace root. Rewrites pipe the existing file content back through the shared LLMAgent and expect a full file replacement wrapped in a code fence. For small edits, replace-text offers deterministic replacements without invoking the LLM.

Testing the CLI

  • Temporary Workspaces — every test places its fixtures under tests/.tmp/<scenario>. Nothing writes to the repository root, and the directory is ignored by Git.
  • Per-test Timeouts — each node:test case explicitly sets a timeout (10–30 seconds) to prevent accidental hangs caused by blocking prompts.
  • Standalone Suites — CLI-specific tests live under tests/cli/gamp and exercise bootstrapping, plan execution, and the fallback pipeline.
  • Manual Runs — to reproduce an individual scenario, invoke node tests/cli/gamp/achiles-cli.test.mjs --test-name-pattern "scenario".
