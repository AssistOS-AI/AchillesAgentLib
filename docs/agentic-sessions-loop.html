Loop Sessions

The loop-based regime implements a bounded reasoning-and-action pattern in which the model repeatedly selects the next admissible tool, observes the result, and continues until the task reaches a completion signal, an interactive pause, or a configured failure boundary.

Deliberative Step Contract

LoopAgentSession is defined in LLMAgents/LoopAgenticSession/LoopAgentSession.mjs. It is initialized over a tools object whose entries expose both a description and a handler. During construction, the runtime rejects any attempt to use the reserved names final_answer or cannot_complete as ordinary tool names and then injects those completion tools into the active tool surface automatically. The session also normalizes its execution options, including bounded defaults for maximum steps per turn, maximum error count, and retry count for expected-value validation.

The constructor may receive initialHistory as ordered, non-empty { role, message } records for the user and assistant roles. It validates and translates those records into the existing internal user and final-answer entries before the first prompt runs; it does not introduce a parallel persisted history schema. When newPrompt() is called, an explicitly supplied model, tags, or reasoningEffort value first replaces the corresponding active session option. The update does not recreate the session, so its conversation history remains intact. The session may then run an optional preparation stage. That preparation stage is itself executed through a bounded loop session and may inject derived context into both the system prompt and the working user prompt before the main turn starts. The core turn is then created with explicit state for prompt text, final answer, retry count, used tools, and step history. The session appends the incoming prompt to its history and enters the running state.

From there, the runtime repeatedly builds planner system instructions through buildAgenticSessionPlannerSystemPrompt() and asks the model for the next decision. Earlier user and assistant turns are derived from the existing session history as role-aware messages, while the current request is sent separately as the final user message. The canonical response is Markdown with tool and prompt sections plus an optional reason section. This remains the primary, non-negotiable, and non-overridable format requested before and after the contextual prompt material, so system-prompt content, tool descriptions, execution context, prior conversation, and the current user request remain planning data rather than alternative formatting instructions.

The runtime parser tolerates common provider variations without guessing external actions. Markdown section names and JSON property names are case-insensitive; separator and camel-case variants such as tool-name, tool_name, toolName, and tool name are equivalent, and the same normalization applies to prompt. Headings, inline labels, bold labels, reordered Markdown fields, JSON objects, and one outer Markdown, text, or JSON fence are accepted. reason may be absent. A declared tool without a prompt receives an empty string. A prompt-only response becomes final_answer, as does a reason-only response, whose reason becomes the final prompt. An explicit final-answer label or non-empty non-JSON prose without recognized planner fields also completes the turn. JSON arrays, primitives, malformed explicitly fenced JSON, JSON objects without recognized planner properties, empty responses, and non-string provider values remain invalid. These recovery rules can complete a turn safely, but only an explicit tool field can initiate a non-terminal tool call.

Intermediate Results and Interactive Continuation

The loop regime is more structured than a naive chat loop because every tool result is stored under a generated symbolic reference. Later prompts may reuse those earlier results through $$variable syntax, which is resolved immediately before the tool handler is called. This allows the planner to depend on explicit prior outputs without inlining large results back into the prompt. The session tracks state across three structures: history (chronological log with result references), toolCalls (metadata-only invocation list for quick lookup), and toolVars (the single source of truth for actual result values). The history records keep this existing domain schema; provider-facing { role, message } records are temporary values derived only for the LLM invocation.

Tool execution itself is mediated. The supervisor is consulted before a tool handler runs. Approval lets the session temporarily set agent.currentSession and invoke the handler; the handler's ordinary result is then stored without adding approval metadata. Denial skips the handler entirely and stores a human-readable result containing the exact tool name, parameters, and supervisor reason under a normal result reference. The planner continues with that result and may select a safe alternative or explain the refusal, but it must not retry the same denied command in the current turn. Supervisor control fields and raw JSON never become conversation text.

Interactive continuation is handled explicitly. If a tool result is a structured payload containing requiresConfirmation or requiresInput, the session moves into awaiting_input state, records the pending tool in history, and returns the interaction message. On the next prompt, the session first checks whether the new user message should continue that pending interaction. To do so, it uses interpretMessage() with the bounded intents accept, cancel, and update, together with a fresh-instruction heuristic. If the new message still belongs to the pending interaction, the session routes it back to the same tool rather than opening a new planner turn.

Note

This pending-input behavior is what makes loop sessions usable as multi-turn governed workflows. In loop-based orchestration and Anthropic Skills, the surrounding subsystem may store the paused session in sessionMemory and resume it on the next user prompt.

Execution Bounds and Return Surface

The loop regime includes several independent stopping and recovery mechanisms. If a tool returns one of the reserved completion wrappers, the session converts that wrapper into the effective final answer. If the caller supplied an expected value for the turn, the session compares the returned completion against that expectation and may retry within a bounded number of attempts. If a tool repeats the same effective invocation and result several times in a row, the session stops rather than allowing trivial loops to continue. If tool errors or planner errors exceed the configured error budget, the session fails explicitly. If the step budget is exhausted, it returns a bounded failure response instead of continuing indefinitely.

Planner failures preserve the same exception path but identify only the remaining invalid response shape, such as an empty response, unsupported JSON, or invalid Markdown. Ordinary non-empty text and JSON objects with recognized planner properties no longer fail parsing because they become normalized decisions. The diagnostic does not compare invalid content with the user request. Tool errors identify unavailable tool names and unknown $$resultRef references, while bounded failures state whether the tool-error budget or planning-step budget was reached.

Loop sessions can also compress long history automatically before each new turn. When estimated token volume crosses a configurable threshold, the runtime sends history entries and resolved result values to the LLM, which returns markdown with summary and keepResultRefs sections. Planner decisions, pending-input classification, preparation, and compression use the active session model; historyCompressionModel can deliberately override it for compression alone. The session replaces old history with one history_summary record, preserves the most recent entries verbatim (default: last 8), and prunes toolVars and toolCalls entries not referenced by the keep list or by recent history. Compression is skipped if a tool is pending in awaiting_input so interactive confirmation flows remain stable.

The return surface is intentionally compact. getLastResult() returns the current last answer. getVariables() returns a small state object containing lastAnswer, the current status, and the number of failed turns. The session also exposes finalizeFailures(), which attempts to convert accumulated failed turns into a cannot_complete result through the reserved completion tool. The regime is therefore adaptive, but its externally visible surface remains narrow.

Cancellation is part of this contract. The loop session links each prompt execution to an internal abort controller and propagates the active signal to planner and interpretation calls. When cancellation is requested, the session moves to interrupted, appends a system interruption entry in history, and returns an interruption answer. The next user prompt clears the interrupted state and resumes normal execution. Tool-level cancellation is cooperative: LLM-facing operations are abortable immediately, while non-interruptible skill handlers may complete before the session stops at the next safe boundary.

const session = await llmAgent.startLoopAgentSession(tools, 'Investigate the request', {
    model: 'plan',
    initialHistory: [
        { role: 'user', message: 'Earlier request' },
        { role: 'assistant', message: 'Earlier answer' },
    ],
    maxStepsPerTurn: 8,
    maxRetriesPerTurn: 3,
    historyCompressionThresholdTokens: 6000,
    historyCompressionKeepRecentEntries: 8,
});

const answer = session.getLastResult();
await session.newPrompt('Continue with deeper analysis', { model: 'deep' });
const state = await session.getVariables();
