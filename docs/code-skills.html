Code Skills

Specification-driven skills that execute stable code generated from explicit requirements.

Vision

Code skills exist to prove the library’s commitment to specification‑driven development. Before any code is generated, the behavior is written down as a specification. That specification becomes the contract, and the generated code becomes the implementation. This is how you keep the LLM aligned with the system’s intent.

The goal is not to avoid the LLM, but to give it a precise plan so it does not drift. A code skill is therefore stable by design, yet still part of the skill system because it is invoked, scoped, and composed like any other skill.

How It Works

A code skill is defined by cskill.md and a set of specification files stored under specs/. When a spec file changes, the system regenerates the corresponding code and writes it under src/. The skill then executes that generated code through the Code Skills subsystem.

Code generation is handled by the built‑in mirror-code-generator skill. It translates specifications into files and performs syntax validation and repair loops when necessary. Details are documented in Built In Skills.

Why It Is a Skill

Even though the execution is deterministic, code skills still benefit from the skill model. They receive the shared LLMAgent and consistent execution context, which keeps behavior aligned across the system and enables the same orchestration patterns used for other skill types.

Folder Structure

Code skills include a descriptor plus a specs directory that feeds the generator. The generated implementation is stored under src/ and should not be edited directly. 

skills/
└── file-system/
    ├── cskill.md
    ├── specs/
    │   └── index.mds
    └── src/
        └── index.mjs

cskill.md Sections

A code skill descriptor is short but strict. These sections define how the skill is invoked and how its output should be interpreted.

  • Summary explains the skill’s purpose at a glance.
  • Input Format defines the required arguments and parsing rules.
  • Output Format specifies how results are returned to callers.
  • Constraints lists boundaries such as path rules, allowed operations, or environment assumptions.

The subsystem expects an Input Format section. If it is missing, the skill is treated as unprepared and will not execute.

Example cskill.md

This example defines a file-system skill with a strict input format and explicit constraints. 

# file-system
Executes file system operations (read, write, delete, copy, list) using nodeJS functions.

## Summary
Executes file system operations (read, write, delete, copy, list) using nodeJS functions. Input must use strict
`operation path` with optional `options:` key-value pairs.

## Input Format
- **operation** (string): Operation type (readFile, writeFile, appendFile, deleteFile, createDirectory,
  listDirectory, fileExists, copyFile, moveFile).
- **path** (string): Target file or directory path.
- **options** (key-value list, optional): `options:` followed by `key: value` pairs.
- **dependsOn** (string, optional): Dependency markers to enforce ordering; ignored by this skill.

Rules:
- First token is always the operation.
- Second token is always the path.
- Optional parameters must be provided after `options:` as `key: value` pairs.
- Values with spaces or newlines must be wrapped in quotes.
- Exception: `content` may be unquoted and may span multiple lines; when unquoted it consumes the rest of the input
  until `dependsOn:` or end of input.
- Supported option keys: `content`, `destination`.
- `dependsOn:` can list multiple refs and is ignored by the parser.

Examples:
- createDirectory ./docs
- writeFile ./test.txt options: content: "hello world"
- writeFile ./specs.md options: content: ## Vision\nLine 1\nLine 2
- readFile ./package.json
- copyFile ./src/file.js options: destination: ./backup/file.js
- writeFile ./specs.md options: content: "..." dependsOn: "$createDir"

## Output Format
- **Type**: `string`
- **Success Example**: "File written successfully" or "{\"exists\":true}" or "[\"file1.txt\",\"file2.txt\"]"
- **Error Example**: "Error: File not found."

## Constraints
- All paths must be relative to project root.
