DBTable Skills

Structured skills for data-centric workflows where consistency matters more than improvisation.

Vision

DBTable skills exist to make data operations predictable and auditable. When your task revolves around records, fields, and constraints, you want the model to follow a strict schema rather than inventing data formats on the fly. DBTable skills formalize that schema so the system can reason about inputs and outputs without ambiguity.

This gives you a consistent contract for CRUD-style work. The skill file defines the table structure and business rules, while the subsystem translates user requests into validated operations.

What It Does

A DBTable skill is defined in tskill.md. The subsystem parses that definition, builds validation and presentation logic, and then uses the shared LLMAgent to guide the interaction. The result is a structured flow that can collect missing fields, validate inputs, and call the underlying database adapter with confidence.

Spec-Driven Code Generation

DBTable skills are not only descriptive; they can generate stable code. The subsystem converts the table definition into a spec file, then invokes the built-in mirror-code-generator to produce a generated module. This keeps behavior consistent across runs while still letting the schema evolve in a controlled way.

The code generation flow is part of the broader built-in skills system, which is documented in Built In Skills.

Why It Is a Skill

DBTable skills are still skills, not static data utilities. They receive the shared LLMAgent so they can interpret user intent, gather missing data, and resolve ambiguous inputs. That flexibility is critical for real-world usage, but it remains bounded by the table definition so outputs stay deterministic and safe.

Folder Structure

A DBTable skill lives in its own folder with a tskill.md descriptor. The generated code is created by the subsystem and should not be edited directly.
