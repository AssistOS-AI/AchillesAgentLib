DBTable Skills

DBTable skills wrap CRUD-style workflows around structured business tables. Instead of hard-coding forms, authors describe schema rules, validation hints, and lookup strategies in tskill.md. The subsystem compiles that descriptor into executable helpers, plans the correct operation (list, view, create, update), gathers missing fields through the interactive executor, and persists results with a registered Persisto client.

Skill Layout

Each DBTable skill lives under .AchillesSkills/<domain>/<table>/ with three important files:

  • tskill.md – the Markdown descriptor that defines the table, field metadata, and behavioural hints.
  • dbtable.generated.mjs – produced by compileDBTableSkill; exports helpers such as prepareRecord and presentRecord.
  • dbtable.generated.meta.json – caches the descriptor hash so recompilation only runs when the author edits tskill.md.

RecursiveSkilledAgent discovers directories that contain tskill.md, assigns them canonical names like projects-dbtable, and delegates preparation and execution to DBTableSkillsSubsystem.

Descriptor Anatomy

A descriptor uses headings to describe both the table overview and each field. Sub-chapters such as Description, Field Name Presenter, Enumerator, Resolver, Validator, Derivator, and PrimaryKey are parsed into the blueprint: 

# Projects
Track internal initiatives and their lifecycle status.

## Table Purpose
Provide lifecycle visibility by listing, creating, or updating projects.

## Field: project_id
### Description
Unique identifier users refer to in chat.
### Field Name Presenter
Project ID
### Resolver
Normalise inputs like `PRJ1` or `Project 1` into the canonical `PRJ-001`.
### PrimaryKey
If empty, generate a PRJ- prefixed identifier with padded counters.

## Field: status
### Required
Status is mandatory when creating or updating records.
### Enumerator
- planned: scheduled but not started
- active: currently in delivery
- complete: fully delivered
### Validator
Reject anything that is not one of planned/active/complete.

Authoring guidelines:

  • Use one ## Field: name section per column. Headings must be unique.
  • Bulleted lists under Aliases or Enumerator are parsed automatically; free-form text also works.
  • Derivator sections describe read-only fields computed via LLM prompts. They are excluded from persistence.
  • Grouping declares cross-table helpers (buildGroupingAccessor) when Persisto exposes richer relations.

Compilation Pipeline

When a descriptor changes, compileDBTableSkill parses it into a blueprint and rewrites dbtable.generated.mjs. The generated module exports:

  • blueprint – normalised table metadata, field ordering, primary keys, and descriptor hash.
  • prepareRecord() – runs resolver instructions on user input so downstream clients receive canonical values.
  • presentRecord() – mirrors presenter instructions to produce human-friendly responses.
  • validatorFunction() – enforces required fields and invokes validator prompts, returning structured issue lists.
  • generatePKValues() – fills blank primary keys via deterministic helpers or LLM instructions.
  • enumeratorFunction() and fieldNamePresenterFunction() – expose options and friendly labels to the UI loop.

These helpers feed directly into runDBTableSkill, ensuring the author’s Markdown stays the single source of truth.

Supported Operations

runDBTableSkill uses the descriptor plus the user prompt to build an operation plan. The flow is:

  1. Heuristics guess the intent (list, show single record, create, update); an LLM prompt refines the plan and extracts filters or values.
  2. For display-table, the subsystem calls tableClient.select() and renders a Markdown table via presentRecord.
  3. For display-record, the plan resolves primary keys, fetches a single record, and prints it as a field/value table.
  4. For create-record or update-record, the subsystem gathers missing fields with the interactive executor, validates input, writes via Persisto, and reports optional fields still missing.

The operation result always includes machine-readable fields (status, issues, rows) plus a Markdown fragment ready to stream back to the user.

Gathering and Validating Fields

DBTable skills reuse the interactive executor so authors inherit the same argument collection experience familiar from iskill.md descriptors. During create/update flows:

  • The subsystem builds a temporary interactive skill where each argument mirrors a field definition (description, enumerator, resolver, validator).
  • Enumerators expose bullet samples or call the LLM instructions to surface searchable option lists.
  • Validators merge hard requirements (missing fields) with structured responses returned by Validator prompts.
  • Optional fields are summarised after execution so operators can decide whether to call the skill again with extra details.

Because the loop is interactive, DBTable skills can confirm destructive actions, cancel gracefully, and surface placeholder detection just like first-class interactive skills.

Persisto Client Contract

DBTable skills never talk to the database directly. Instead, register an adapter that exposes the CRUD surface area for each table: 

import { registerPersistoClient } from 'achillesAgentLib/DBTableSkillsSubsystem';
import { Persisto } from './persisto.js';

registerPersistoClient({
    getProjects: (id) => Persisto.fetchProject(id),
    updateProjects: (record) => Persisto.saveProject(record),
    createProjects: (record) => Persisto.insertProject(record),
    select: (tableName, filters, options) => Persisto.query(tableName, filters, options),
});

Method names follow predictable patterns (get${PascalCase(tableName)}, update${PascalCase(tableName)}, create${PascalCase(tableName)}). Optional helpers such as buildGroupingAccessor or buildRelationAccessor can call richer Persisto APIs when descriptors declare Grouping data.

LLM Collaboration

All free-form instructions inside tskill.md are fed to LLMAgent at runtime:

  • Enumerators: either return the bullet list directly or ask the model to build options dynamically.
  • Resolvers and Presenters: prompts teach the model how to normalise inbound values and how to display them back.
  • Validators: produce JSON payloads such as { "issues": [{ "field": "status", "message": "..."}]} that the UI can map to prompts.
  • Derivators: let authors compute read-only insights (for example, “overall health”) every time a record is displayed.

Because the descriptor narrates both directions (resolver vs presenter), DBTable skills remain reversible—the system always knows how to store a value and how to explain it.

Testing and Local Development

Use tests/runDBTableTests.js to exercise DBTable flows in isolation. The helper tests/dbtableSkills/helpers/runDBTableScenario.mjs wires a MockPersistoClient plus a RecursiveSkilledAgent so you can script realistic prompts in Node tests. A typical workflow is:

  1. Author or update tskill.md and run the tests; the compiler regenerates dbtable.generated.mjs automatically.
  2. Use runDBTableScenario to seed fake table data and simulate user responses.
  3. Assert on execution.result.record, execution.result.rows, or execution.result.issues.

This keeps DBTable skills aligned with the rest of the testable architecture: Markdown describes intent, generated modules implement it, and automated scenarios guarantee nothing regresses.
